## Tantangan dan Proses Penyelesaian

Tantangan teknis paling signifikan yang saya hadapi terjadi pada fase kritis otomatisasi *Continuous Deployment* ke platform *cloud* Railway. Kegagalan berulang pada *CD Pipeline* sempat melumpuhkan alur kerja akibat isu token autentikasi yang kedaluwarsa serta galat deteksi komponen layanan (*Service not found*). Karakteristik arsitektur *microservices* yang terdekomposisi menuntut ketelitian tinggi dalam sinkronisasi variabel lingkungan (*environment variables*) dan *secrets* lintas lingkungan kerja. Akar penyebab permasalahan ini berhasil diidentifikasi setelah melakukan analisis mendalam terhadap log GitHub Actions, di mana terjadi ketidaksesuaian penamaan parameter layanan operasional antara manifes lokal dengan dasbor *cloud*. Penanganan masalah dilakukan melalui restrukturisasi variabel rahasia pada GitHub Secrets dan sinkronisasi flag spesifik lingkungan produksi pada file konfigurasi alur kerja YAML.

Hambatan berikutnya muncul pada tahap *Health Check* pasca-deployment, di mana pipa otomasi mendeteksi kegagalan sistemik berupa kode respons HTTP 301 (*Moved Permanently*) dan ketidaksesuaian status pembacaan bodi JSON. Proses investigasi menemukan adanya mekanisme internal platform *cloud* yang secara otomatis mengalihkan lalu lintas data HTTP tidak aman ke HTTPS, sementara skrip pengujian awal secara keliru memvalidasi string respons statis "ok" alih-alih "healthy" yang dikirim oleh FastAPI. Penyelesaian kendala ini membutuhkan pemikiran analitis yang matang melalui implementasi flag pengalihan lokasi (`-L`) pada instruksi `curl` serta penyelarasan skrip Python pembaca JSON untuk memvalidasi indikator kesehatan riil dari basis data yang terhubung. Kegagalan mitigasi terhadap isu visibilitas sistem (*observability*) dan penanganan kegagalan (*failure handling*) ini sejak dini berpotensi memicu kondisi *false negative* pada pipeline, yang pada akhirnya dapat meloloskan deployment cacat ke tangan pengguna akhir dan merusak reputasi keandalan layanan.

## Pelajaran yang Diperoleh

Selama mengintegrasikan kode kelompok ke dalam pipeline otomatis dan menyebarkannya ke platform *cloud*, saya menghadapi beberapa tantangan teknis serius yang menuntut proses *troubleshooting* mendalam:

- **Kendala Autentikasi dan Sinkronisasi Token (Masalah Token Kedaluwarsa):**
  Saat pertama kali menjalankan pipa CD ke platform Railway, proses *deployment* langsung macet total akibat token akses yang tidak valid atau kedaluwarsa. Karakteristik sistem *microservices* menuntut ketelitian tinggi dalam mencocokkan kredensial rahasia kelompok. Masalah ini diselesaikan dengan cara me-regenerasi token proyek baru dari dasbor Railway dan memperbarui konfigurasi pada GitHub Secrets agar pipa otomasi mendapatkan izin akses penuh kembali.
- **Galat Identifikasi Komponen (Eror *Service Not Found*):**
  Setelah masalah token selesai, pipeline kembali berhenti karena Railway tidak mengenali komponen layanan *backend* dan *frontend*. Setelah mengulik log di GitHub Actions, saya menemukan akar masalahnya berupa ketidaksesuaian penamaan layanan antara file YAML konfigurasi lokal dengan nama layanan rill yang kami daftarkan di dasbor Railway kelompok kami.
- **Pengalihan Protokol Komunikasi Terbuka (Eror HTTP Status 301):**
  Pada tahap pengujian kesehatan (*Health Check*), pipa otomatis mendeteksi kegagalan karena mendapatkan respons status HTTP 301 (*Moved Permanently*). Investigasi log menunjukkan bahwa platform Railway otomatis mengalihkan semua lalu lintas data HTTP biasa ke protokol HTTPS yang aman, sementara skrip pengujian kami masih menembak URL HTTP biasa.
- **Ketidaksesuaian Validasi Nilai Struktur Data (Eror Gagal Parsing JSON):**
  Tantangan terakhir muncul karena skrip otomatis bawaan mencari kata kunci status "ok" untuk menyatakan aplikasi sehat, sementara fungsi backend FastAPI kelompok kami mengirimkan data berformat "status": "healthy". Ketidakcocokan string ini membuat pipa otomatis terus membaca status *false-negative* (dikira eror padahal aplikasi berjalan normal). Saya membereskannya dengan menyesuaikan logika pembanding di dalam skrip Python pembaca JSON agar selaras dengan keluaran riil dari backend.

Jika tantangan-tantangan operasional di atas tidak segera dimitigasi dengan baik, akibatnya alur kerja tim akan terhambat lama karena tidak bisa merilis fitur baru ke platform *cloud*, serta berisiko meloloskan aplikasi yang tidak sehat ke tangan pengguna akhir.

## Kesimpulan

Dari semua dinamika dan proses naik-turun yang dilewati selama mempraktikkan modul ini, saya menyimpulkan kalau peran seorang Lead DevOps Engineer punya pengaruh yang sangat besar terhadap kecepatan rilis dan kestabilan sebuah aplikasi. Arahan dosen untuk merancang kontainerisasi dengan Docker dan otomatisasi CI/CD sejak awal terbukti menjadi investasi waktu yang sangat menguntungkan saat tim mulai menggabungkan seluruh kode (full-stack integration). Lewat proses troubleshooting eror health check dan konfigurasi cloud kemarin, kemampuan analisis dan trobelshuting saya dalam menangani sistem terdistribusi jadi jauh lebih tajam. Pengalaman berharga dari modul ini menjadi modal yang kuat bagi saya untuk menghadapi proyek infrastruktur modern ke depannya, dengan keyakinan penuh bahwa otomatisasi yang matang adalah kunci utama di balik aplikasi cloud yang sukses dan tahan banting.
